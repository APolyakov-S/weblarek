## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

- Model - слой данных, отвечает за хранение и изменение данных.
- View - слой представления, отвечает за отображение данных на странице.
- Presenter - презентер содержит основную логику приложения и отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component

Является базовым классом для всех компонентов интерфейса. Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`

#### Класс Api

Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter

Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` - хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.

## Данные

В приложении определены две основные сущности, описывающие бизнес-логику: **Товар** и **Покупатель**.

### Интерфейсы данных

**`IProduct`** — описывает структуру товара, полученного с сервера или используемого в приложении.

```typescript
interface IProduct {
  id: string;          // Уникальный идентификатор товара
  description: string; // Описание товара
  image: string;       // URL-адрес изображения товара
  title: string;       // Название товара
  category: string;    // Категория товара
  price: number | null;// Цена товара. null означает, что товар нельзя купить
}
```

**`IBuyer`** — описывает данные покупателя, необходимые для оформления заказа.

```typescript
type TPayment = 'card' | 'cash';

interface IBuyer {
  payment: TPayment | null;   // Способ оплаты: картой ('card') или наличными ('cash')
  email: string;              // Электронная почта
  phone: string;              // Номер телефона
  address: string;            // Адрес доставки
}
```

## Модели данных

Модели данных отвечают за хранение, изменение и предоставление данных приложения. Все модели реализуют событийный подход: при изменении данных они генерируют события, на которые подписываются другие части приложения.

### Класс `CatalogModel`

**Назначение:** управляет каталогом товаров. Хранит список всех доступных товаров и текущий выбранный товар для детального просмотра.

**Конструктор:**  
`constructor(events: EventEmitter)`

**Поля класса:**
- `private items: IProduct[]` — массив всех товаров в каталоге.
- `private preview: IProduct | null` — товар, выбранный для просмотра деталей.

**Методы класса:**
- `public setItems(items: IProduct[]): void` — сохраняет массив товаров. Генерирует событие `catalog:changed`.
- `public getItems(): IProduct[]` — возвращает массив всех товаров.
- `public getItem(id: string): IProduct | undefined` — возвращает товар по id.
- `public setPreview(item: IProduct): void` — сохраняет товар для просмотра. Генерирует событие `preview:changed`.
- `public getPreview(): IProduct | null` — возвращает текущий выбранный товар.

### Класс `CartModel`

**Назначение:** управляет корзиной покупателя.

**Конструктор:**  
`constructor(events: EventEmitter)`

**Поля класса:**
- `private items: IProduct[]` — массив товаров в корзине.

**Методы класса:**
- `public getItems(): IProduct[]` — возвращает массив товаров в корзине.
- `public addItem(item: IProduct): void` — добавляет товар. Генерирует событие `cart:changed`.
- `public removeItem(itemId: string): void` — удаляет товар по id. Генерирует событие `cart:changed`.
- `public clear(): void` — очищает корзину. Генерирует событие `cart:changed`.
- `public getTotal(): number` — возвращает общую стоимость.
- `public getCount(): number` — возвращает количество товаров.
- `public hasItem(id: string): boolean` — проверяет наличие товара в корзине.

### Класс `BuyerModel`

**Назначение:** управляет данными покупателя при оформлении заказа.

**Конструктор:**  
`constructor(events: EventEmitter)`

**Поля класса:**
- `private payment: TPayment | null` — выбранный способ оплаты
- `private address: string` — адрес доставки
- `private email: string` — электронная почта
- `private phone: string` — номер телефона

**Методы класса:**
- `public setPayment(payment: TPayment): void` — сохраняет способ оплаты. Генерирует событие `buyer:changed`.
- `public setAddress(address: string): void` — сохраняет адрес. Генерирует событие `buyer:changed`.
- `public setEmail(email: string): void` — сохраняет email. Генерирует событие `buyer:changed`.
- `public setPhone(phone: string): void` — сохраняет телефон. Генерирует событие `buyer:changed`.
- `public getBuyerData(): IBuyer` — возвращает все данные покупателя.
- `public clear(): void` — очищает данные. Генерирует событие `buyer:changed`.
- `public validate(): TValidationErrors` — валидирует данные. Возвращает объект с ошибками.

## Слой коммуникации

Класс `WebLarekApi` отвечает за взаимодействие с сервером API.

### Класс `WebLarekApi`

**Назначение:** обеспечивает получение данных о товарах с сервера и отправку данных о заказе.

**Конструктор:**  
`constructor(api: IApi)`

**Поля класса:**
- `private api: IApi` — экземпляр класса Api для выполнения запросов

**Методы класса:**
- `public getProducts(): Promise<IProductsResponse>` — выполняет GET запрос на эндпоинт `/product`. Возвращает промис с объектом, содержащим массив товаров.
- `public postOrder(order: IOrderData): Promise<IOrderResult>` — выполняет POST запрос на эндпоинт `/order`. Принимает данные заказа и возвращает промис с результатом.

## Слой представления (View)

Классы представления отвечают за отображение данных на странице и взаимодействие с пользователем. Все классы наследуются от базового класса `Component<T>`.

### Базовый класс `Card<T>`

**Назначение:** базовый класс для всех карточек товаров.

**Поля класса:**
- `protected titleElement: HTMLElement` — элемент для заголовка
- `protected priceElement: HTMLElement` — элемент для цены
- `protected imageElement?: HTMLImageElement` — элемент для изображения
- `protected categoryElement?: HTMLElement` — элемент для категории
- `protected buttonElement?: HTMLButtonElement` — кнопка взаимодействия

**Методы класса:**
- `set title(value: string)` — устанавливает заголовок
- `set price(value: number | null)` — устанавливает цену
- `set image(value: string)` — устанавливает изображение
- `set category(value: string)` — устанавливает категорию
- `set buttonText(value: string)` — устанавливает текст кнопки
- `set buttonDisabled(value: boolean)` — блокирует/разблокирует кнопку

### Класс `CardCatalog`

**Назначение:** карточка товара для отображения в каталоге. Наследуется от `Card<T>`. При клике генерирует событие `card:select`.

### Класс `CardPreview`

**Назначение:** карточка товара для отображения в модальном окне. Наследуется от `Card<T>`. Дополнительно содержит поле `description`.

### Класс `CardBasket`

**Назначение:** карточка товара для отображения в корзине. Наследуется от `Component<T>`. При клике на кнопку "Удалить" генерирует событие `basket:remove`.

### Базовый класс `Form<T>`

**Назначение:** базовый класс для всех форм.

**Поля класса:**
- `protected formElement: HTMLFormElement` — элемент формы
- `protected submitButton: HTMLButtonElement` — кнопка отправки
- `protected errorsElement: HTMLElement` — элемент для отображения ошибок

**Методы класса:**
- `set valid(value: boolean)` — включает/выключает кнопку отправки
- `set errors(value: Partial<Record<keyof T, string>>)` — отображает ошибки

### Класс `OrderForm`

**Назначение:** форма для выбора способа оплаты и ввода адреса. Наследуется от `Form<IOrderFormData>`. При отправке генерирует событие `order:submit`.

### Класс `ContactsForm`

**Назначение:** форма для ввода контактных данных. Наследуется от `Form<IContactsFormData>`. При отправке генерирует событие `contacts:submit`.

### Класс `Modal`

**Назначение:** управление модальным окном.

**Методы класса:**
- `open(): void` — открывает модальное окно
- `close(): void` — закрывает модальное окно
- `set content(value: HTMLElement)` — устанавливает контент

### Класс `Basket`

**Назначение:** отображение корзины. При клике на кнопку оформления генерирует событие `basket:checkout`.

### Класс `Success`

**Назначение:** отображение сообщения об успешном заказе. При клике на кнопку генерирует событие `success:closed`.

### Класс `Page`

**Назначение:** управление основным содержимым страницы.

**Методы класса:**
- `set catalog(value: HTMLElement[])` — устанавливает карточки в галерею
- `set counter(value: number)` — устанавливает значение счётчика корзины
- `set locked(value: boolean)` — блокирует/разблокирует прокрутку страницы

## Презентер

Презентер реализован в основном скрипте приложения `main.ts`. Он отвечает за связь между моделями данных и представлениями, обрабатывает все события и управляет логикой приложения.

### Подход к реализации

Презентер не вынесен в отдельный класс, а написан непосредственно в `main.ts`. Это упрощает код, так как приложение имеет одну страницу и не требует сложной маршрутизации. Весь код презентера состоит из:
- инициализации компонентов (модели, представления, API)
- подписки на события от моделей и представлений
- обработчиков событий, которые обновляют данные и перерисовывают интерфейс

### Принципы работы

1. Презентер не генерирует события, только обрабатывает их
2. При изменении данных в моделях генерируются события, презентер обновляет представления
3. При действиях пользователя представления генерируют события, презентер обновляет модели
4. Представления перерисовываются только при изменении данных или открытии модального окна

### Список обрабатываемых событий

**От моделей данных:**
- `catalog:changed` - обновление каталога товаров
- `preview:changed` - выбор товара для просмотра
- `cart:changed` - изменение корзины
- `buyer:changed` - изменение данных покупателя

**От представлений:**
- `card:select` - клик по карточке товара в каталоге
- `basket:remove` - удаление товара из корзины
- `basket:checkout` - оформление заказа из корзины
- `order:submit` - отправка формы заказа
- `contacts:submit` - отправка формы контактов
- `modal:closed` - закрытие модального окна
- `success:closed` - закрытие окна успешного заказа

### Основные сценарии работы

**Загрузка каталога:** получение данных с сервера -> сохранение в модель -> генерация события catalog:changed -> отрисовка карточек

**Открытие карточки:** клик на карточку -> генерация события card:select -> создание карточки предпросмотра -> открытие модального окна

**Добавление в корзину:** клик на кнопку "В корзину" -> добавление в модель -> генерация события cart:changed -> обновление корзины и счетчика

**Оформление заказа:** открытие корзины -> форма заказа -> форма контактов -> валидация -> отправка на сервер -> очистка корзины -> сообщение об успехе
